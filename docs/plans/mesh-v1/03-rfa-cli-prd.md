# PRD — `rfa` CLI (Day-1 Mandatory)

**Status:** Open
**Companion to:** `00-mesh-rebuild-prd.md` + `02-topology-ui-prd.md`
**Ships in:** Sprint 0; no deferral.

---

## 1. The mandate

Every operation the topology UI supports MUST also be scriptable via the `rfa` CLI. If an operator can do it in the UI, they can do it in CI, in a shell loop, in a soak-test driver.

The CLI is not an afterthought. It is the second authoritative control surface. UI and CLI talk to the same backend (the topology UI process's REST API + mesh client).

## 2. Command surface (Sprint 0)

```bash
# Node lifecycle
rfa mesh node add --type <gateway|broker|compute|schema> [--name <slug>] [--bind <addr>]
rfa mesh node remove <node-id-or-name>
rfa mesh node list [--type <type>]
rfa mesh node describe <node-id-or-name>
rfa mesh node logs <node-id-or-name> [--follow] [--tail <N>]
rfa mesh node spans <node-id-or-name> [--since <duration>] [--name <span-name>]

# Topology
rfa mesh topology show [--format text|json|dot]
rfa mesh topology watch              # streams updates to stdout, like `kubectl get pods -w`

# Health
rfa mesh status                      # quick summary: N nodes by type, X edges, Y stale, Z partitioned
rfa mesh wait-converged --timeout <duration>   # blocks until topology stable; useful in CI
```

## 3. Command surface (Sprint 1 — chaos)

```bash
rfa mesh chaos partition <node-a> <node-b> [--duration <duration>]
rfa mesh chaos flap <edge> --interval <duration> [--count <N>]
rfa mesh chaos kill <node-id> [--restart-after <duration>]
rfa mesh chaos nat-shift <node-id>
rfa mesh chaos soak --duration 24h [--seed <random-seed>]
rfa mesh chaos catalog               # lists all available chaos primitives
```

## 4. Command surface (Sprint 2 — multi-mesh)

```bash
rfa mesh add --name <mesh-id> [--relay <url>]
rfa mesh list
rfa mesh peer --from <mesh-a> --to <mesh-b>
rfa mesh unpeer --from <mesh-a> --to <mesh-b>
rfa mesh topology show --multi-mesh
```

## 5. Implementation

`rfa` is a single binary in `crates/rfa/`. It is a thin REST client targeting the `rafka-topology-ui` process (default `http://localhost:19090`). Each subcommand maps 1:1 to a REST call.

This means:
- Zero new mesh primitives in the CLI itself — all logic lives in `rafka-topology-ui`
- CLI output is JSON parsed from REST responses, formatted via `clap` + `serde`
- `--format json` for every command for scripting
- `--format text` (default) for human-readable
- `--format dot` (topology only) for Graphviz piping

## 6. Why the CLI calls the UI backend (not the mesh directly)

Two reasons:

1. **Single control surface.** UI and CLI hitting the same backend means they can't drift. If a chaos primitive works in the UI but not the CLI (or vice versa), it's the backend's bug, not a duplicated implementation.

2. **Idempotent state machine.** The UI backend owns subprocess lifecycle (PIDs, data dirs, log handles). Having two independent process-manager implementations (one in UI, one in CLI) is a guaranteed source of "killed via CLI but UI shows alive" bugs.

The cost: if the UI process is down, the CLI doesn't work. Acceptable for a dev/chaos-test tool. In production, neither matters (production uses k8s/systemd).

## 7. Example workflows

**Local dev — spin up a small mesh:**

```bash
rfa mesh node add --type gateway --name gw-1
rfa mesh node add --type broker --name br-1
rfa mesh node add --type broker --name br-2
rfa mesh node add --type compute --name cp-1
rfa mesh wait-converged --timeout 30s
rfa mesh topology show --format dot | dot -Tpng > topology.png
```

**Chaos run in CI:**

```bash
rfa mesh chaos soak --duration 1h --seed $CI_RUN_ID &
SOAK_PID=$!
rfa mesh status --json | jq '.healthy_count' > pre.txt
sleep 1h
wait $SOAK_PID
rfa mesh status --json | jq '.healthy_count' > post.txt
diff pre.txt post.txt || exit 1
```

**Debugging a flaky test:**

```bash
rfa mesh node logs gw-1 --tail 100
rfa mesh node spans gw-1 --since 5m --name rafka.mesh.peer.staleness_timeout
rfa mesh node describe br-1
```

## 8. Acceptance criteria (Sprint 0)

1. `cargo install --path crates/rfa` installs the binary
2. `rfa mesh node add --type gateway` returns the new node's `EndpointId` within 5s
3. `rfa mesh node list` shows the just-added node
4. `rfa mesh topology show` prints the same topology the UI is showing (text or json)
5. `rfa mesh wait-converged --timeout 10s` blocks until topology is stable, then exits 0
6. `rfa mesh node remove <id>` terminates the subprocess; node disappears from `rfa mesh node list` within 10s
7. Every command supports `--format json` for scripting
8. Every command has `--help` text generated by clap

## 9. Non-goals (Sprint 0)

- **No interactive shell mode.** Stateless one-shot commands only.
- **No remote mesh management.** Localhost target only. Production node mgmt is via k8s/systemd.
- **No config files.** All options via flags or env vars (CLAUDE.md KISS).
- **No authentication.** Localhost dev tool. Same as the UI.
- **No data-plane ops** (produce, fetch, etc.). Those come when the app layer comes back.
